# AutoApplyer AI - Architecture Overview

**Version**: 1.0  
**Date**: 2025-01-15  
**Purpose**: High-level architecture documentation for onboarding engineers in 10 minutes

---

## System Overview

AutoApplyer AI is a DWP-facing SaaS platform that automates Universal Credit job search and compliance for claimants and work coaches. The system uses browser automation (Playwright) to apply to jobs on platforms like Indeed, Greenhouse, and Lever, while providing dashboards for claimants, work coaches, and DWP administrators.

---

## Backend Architecture

### Entry Point
- **File**: `autoapply/server.py`
- **Framework**: FastAPI (Python 3.10+)
- **Port**: 8000 (default)
- **Purpose**: Main HTTP server handling authentication, API endpoints, and serving the React SPA

### Storage Layer

#### SQLite Database (`data/auth.db`)
- **Purpose**: User authentication and authorization
- **Schema**: Managed by `autoapply/auth_store.py`
- **Tables**: Users (email, password hash, role, display_name, metadata)
- **Roles**: `claimant`, `coach`, `admin`

#### CSV Logs (`data/logs.csv`)
- **Purpose**: Application activity audit trail
- **Format**: CSV with headers: `ts`, `site`, `job_title`, `company`, `url`, `status`, `notes`
- **Writer**: `autoapply/core/storage.py::log()`
- **Reader**: `autoapply/server.py` (for dashboards and exports)

#### JSON Artifacts
- **Jobs Queue**: `data/jobs_queue.json` - Discovered jobs awaiting review/approval
- **Analytics Log**: `data/analytics.jsonl` - Frontend analytics events (JSONL format)
- **Artifacts**: `data/artifacts/` - Screenshots and debugging artifacts from automation runs

#### YAML Configuration (`config.yaml`)
- **Purpose**: Per-claimant job search configuration
- **Schema**: Defined by `autoapply/core/config.py` (Pydantic models)
- **Sections**: Account info, job searches, limits, filters, CV paths, cover letter templates
- **Note**: One `config.yaml` per installation (multi-tenant support would require per-user configs)

### Automation Engine

#### Playwright Browser Automation
- **Entry Point**: `autoapply/run.py::main()`
- **Browser**: Chromium (via Playwright)
- **Profile**: Persistent browser profile stored in `profiles/default/` (saves session cookies, no password storage)
- **Adapters**: Platform-specific automation logic
  - `autoapply/adapters/indeed.py` - Indeed Easy Apply
  - `autoapply/adapters/greenhouse.py` - Greenhouse application forms
  - `autoapply/adapters/lever.py` - Lever application forms
- **Core Modules**:
  - `autoapply/core/browser.py` - Browser session management
  - `autoapply/core/router.py` - Job discovery and navigation
  - `autoapply/core/autofill.py` - Form filling logic
  - `autoapply/core/tailor.py` - CV/cover letter customization

#### Job Logs Location
- **Primary Log**: `data/logs.csv` (one row per application attempt)
- **Artifacts**: `data/artifacts/{timestamp}/` (screenshots, HTML snapshots)
- **Access**: Via FastAPI endpoints (`/api/claimant/status`, `/api/staff/work-coach/claimants/{id}`)

## Roles and User Flows

### 1. Claimant (Universal Credit Claimant)

**Access**: `/app/*` routes  
**App Shell**: `ClaimantAppShell` (sidebar navigation)

#### Key Flows

**Onboarding**
1. New claimant signs up → `/app/onboarding`
2. Multi-step form collects: profile, skills, job preferences, automation settings
3. On completion → `/app/onboarding/confirmation` → `/app/dashboard`
4. Backend creates profile and initial `config.yaml`

**Dashboard**
- **Route**: `/app/dashboard`
- **Shows**: Compliance summary (applications this week), activity timeline, upcoming tasks
- **Data Source**: `/api/claimant/status` (returns stats + recent activity log)

**Job Search & Review**
- **Route**: `/app/jobs` (placeholder - needs implementation)
- **Route**: `/app/applications/review` (batch review before applying)
- **Purpose**: Review discovered jobs, approve/reject applications
- **Data Source**: `/api/claimant/jobs` (returns jobs from queue)

**Application History**
- **Route**: `/app/applications` (placeholder - needs implementation)
- **Purpose**: View full history of all applications with filtering

**Settings**
- **Route**: `/app/settings` (placeholder - needs implementation)
- **Purpose**: Update profile, CV, job search preferences
- **Backend**: `PUT /api/claimant/profile`

## Authentication & Authorization

### Session Management
- **Method**: HTTP-only cookies (`autoapply_session`)
- **Duration**: 7 days
- **Secret**: Environment variable `AUTOAPPLYER_SESSION_SECRET` (or default for dev)
- **Implementation**: `autoapply/auth.py` (signed tokens with HMAC-SHA256)

### Role-Based Access Control
- **Backend**: `check_role()` function in `autoapply/auth.py` enforces role requirements on API endpoints
- **Frontend**: `ProtectedRoute` component in `src/components/routing/ProtectedRoute.tsx` checks roles client-side
- **Roles**: `claimant`, `coach`, `admin` (defined in `autoapply/auth_store.py::Role` enum)

## API Architecture

### Authentication Endpoints
- `POST /auth/login` - Login, returns session cookie
- `POST /auth/logout` - Logout, clears session cookie
- `POST /auth/register` - Register new user (dev/seed only)
- `GET /auth/me` - Get current authenticated user info

### Claimant API Endpoints
- `GET /api/claimant/status` - Get claimant status and activity data
- `GET /api/claimant/jobs` - Get job listings from queue (with filters)
- `POST /api/claimant/applications/batch` - Batch approve/reject applications
- `PUT /api/claimant/profile` - Update claimant profile and preferences

### Staff API Endpoints (Work Coach)
- `GET /api/staff/work-coach/claimants` - List assigned claimants (with filters)
- `GET /api/staff/work-coach/claimants/{claimant_id}` - Get detailed claimant record
- `GET /api/staff/work-coach/reports/export` - Export CSV/PDF report

### Staff API Endpoints (DWP/Admin)
- `GET /api/staff/dwp/metrics` - Get aggregated regional metrics
- `GET /api/staff/regions` - Get list of regions
- `GET /api/staff/jobcentres` - Get list of jobcentres (optionally filtered by region)

### Analytics
- `POST /api/analytics` - Track analytics events from frontend (logged to JSONL)

### Legacy Endpoints (Jinja2 Templates)
- `GET /setup` - Legacy setup form (Jinja template)
- `POST /setup` - Save configuration (updates `config.yaml`)
- `POST /run` - Trigger automation engine (background thread)
- `GET /status` - Legacy status page (Jinja template)

## Data Flow Examples

### Example 1: Claimant Views Dashboard
1. User navigates to `/app/dashboard`
2. React Router renders `ClaimantDashboardContainer`
3. Component calls `GET /api/claimant/status` via `apiClient.ts`
4. Backend reads `data/logs.csv`, aggregates stats, returns JSON
5. Frontend displays compliance summary and activity timeline

### Example 2: Automation Run
1. User triggers automation (via `/run` endpoint or UI button)
2. Backend spawns background thread running `autoapply/run.py::main()`
3. Playwright opens browser, navigates to job sites
4. For each job: applies (or skips), logs to `data/logs.csv`
5. Frontend polls `/api/claimant/status` to show progress

### Example 3: Work Coach Views Claimant
1. Coach navigates to `/staff/work-coach`
2. Dashboard calls `GET /api/staff/work-coach/claimants`
3. Backend reads `data/logs.csv`, groups by user, calculates compliance metrics
4. Coach clicks on claimant → navigates to `/staff/work-coach/claimants/{id}`
5. Detail page calls `GET /api/staff/work-coach/claimants/{claimant_id}`
6. Backend returns full activity log and profile data

## Deployment Considerations

### Current State (Development)
- Single-server deployment (FastAPI + React SPA)
- SQLite for auth (single file, no migrations)
- CSV for logs (append-only, no concurrent writes)
- One `config.yaml` per installation

### Pilot Scale (5,000 Claimants)
- **Architecture**: Can scale horizontally with load balancer
- **Database**: SQLite may need migration to PostgreSQL for concurrent writes
- **Storage**: CSV logs may need migration to database table for query performance
- **Config**: Need per-user config storage (database or file system per user)

### Production Scale (National)
- **Database**: PostgreSQL with connection pooling
- **Job Queue**: Redis or dedicated job queue (Celery, RQ)
- **Caching**: Redis for job listings and metrics
- **Real-time**: WebSockets for automation status updates
- **File Storage**: S3 or equivalent for CVs, artifacts
- **Monitoring**: Structured logging, error tracking (Sentry), metrics (Prometheus)

---

## Key Design Decisions

1. **No Password Storage**: Browser profile stores session cookies, not passwords
2. **CSV Logs**: Simple, append-only format suitable for audit trails and Excel export
3. **Single Config File**: One `config.yaml` per installation (simplifies deployment, but limits multi-tenancy)
4. **React SPA**: Client-side routing with FastAPI serving static files
5. **Role-Based Shells**: Separate app shells (`ClaimantAppShell`, `CoachAppShell`, `AdminAppShell`) for clear separation
6. **Design System**: Centralized tokens in `src/design/system.ts` for consistency

## Future Considerations

- **Multi-tenancy**: Per-user config storage, user isolation
- **Job Queue System**: Dedicated queue (Redis/Celery) for discovered jobs
- **Real-time Updates**: WebSockets for automation progress
- **Database Migration**: SQLite → PostgreSQL for scale
- **Mobile Apps**: PWA support (manifest, service worker) for installable web app
- **Advanced Analytics**: Job matching scores, success rate predictions
